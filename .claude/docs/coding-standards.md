# 编码规范

本文件定义 Software Engineering Studios 中所有 Agent 必须遵守的编码规范、设计文档规范、测试规范和 CI/CD 规则。

## 1. 编码原则

### 1.1 公共 API 文档注释

所有公共 API 必须包含完整的文档注释，确保可自动生成 API 文档。

- 每个公共函数/方法/类必须包含用途说明
- 参数必须标注类型、含义和约束
- 返回值必须标注类型和含义
- 异常/错误必须列出可能的类型和触发条件
- 示例代码应附在复杂 API 的注释中

示例（TypeScript）：
```typescript
/**
 * 创建用户会话
 * @param userId - 用户唯一标识，必须为非空字符串
 * @param options - 会话选项
 * @returns 会话令牌对象
 * @throws {InvalidUserIdError} 当 userId 为空或格式不合法时抛出
 * @throws {SessionLimitExceededError} 当用户并发会话超过上限时抛出
 */
```

### 1.2 每个系统需 ADR

每个系统/模块必须至少有一份架构决策记录（ADR），记录关键技术选型和架构决策。

- ADR 存放于 `docs/architecture/decisions/` 目录
- 命名格式：`ADR-NNN-简短标题.md`（如 `ADR-001-选择-PostgreSQL-作为主数据库.md`）
- 至少 3 条 ADR 方可通过架构审查门禁
- 详见 `/architecture-decision` 命令

### 1.3 配置数据驱动不硬编码

所有环境相关配置、业务参数、阈值等必须数据驱动，禁止硬编码。

- 配置文件存放于 `src/config/` 或 `config/` 目录
- 环境变量通过 `.env` 文件管理，禁止提交到版本控制
- 业务可配置参数应支持运行时调整
- 不同环境（开发/测试/预发布/生产）使用独立配置

### 1.4 依赖注入优于单例

优先使用依赖注入（DI）模式，避免单例模式带来的测试困难和耦合问题。

- 通过构造函数注入依赖
- 使用 IoC 容器管理依赖生命周期
- 单例仅在以下场景允许：
  - 无状态的工具类
  - 基础设施层（如日志、配置）
  - 经 `chief-architect` 审批确认

### 1.5 Conventional Commits

所有 Git 提交信息必须遵循 Conventional Commits 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）：**

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| `feat` | 新功能 | `feat(auth): 添加 OAuth2 登录支持` |
| `fix` | 缺陷修复 | `fix(api): 修复分页参数 off-by-one 错误` |
| `docs` | 文档变更 | `docs(api): 更新用户接口文档` |
| `refactor` | 重构（不改变外部行为） | `refactor(auth): 提取令牌验证逻辑` |
| `test` | 测试相关 | `test(user): 添加用户注册单元测试` |
| `chore` | 构建/工具/杂项 | `chore(deps): 升级依赖至最新版本` |

**规则：**
- `subject` 使用简体中文，不超过 50 字
- `body` 说明变更原因和背景
- `footer` 标注 BREAKING CHANGE 或关联 issue
- 提交前由 `validate-commit.sh` Hook 自动校验

### 1.6 验证驱动开发

采用验证驱动开发（Verification-Driven Development），每个功能在实现前必须明确验证方式。

- 每个用户故事必须包含验收标准
- 实现前先定义测试用例
- 测试证据必须可追溯至需求
- 详见第 4 节测试规范

## 2. 设计文档规范

每个系统/模块的设计文档必须包含以下 8 个必需章节：

### 2.1 概述

- 系统名称与定位
- 核心职责
- 在整体架构中的位置
- 相关利益相关者

### 2.2 用户故事

- 以用户视角描述使用场景
- 格式：作为 [角色]，我想要 [功能]，以便 [价值]
- 覆盖主要使用路径和边缘场景

### 2.3 详细规则

- 业务规则与约束
- 状态转换与生命周期
- 权限与访问控制
- 数据校验规则

### 2.4 数据模型

- 实体定义与关系
- 字段类型与约束
- 索引策略
- 数据迁移方案

### 2.5 边界情况

- 极端输入处理
- 并发与竞态条件
- 失败与回滚场景
- 性能边界与降级策略

### 2.6 依赖关系

- 上游依赖（本系统依赖的其他系统）
- 下游依赖（依赖本系统的其他系统）
- 外部服务依赖
- 版本兼容性要求

### 2.7 可配置参数

- 所有可配置项及其默认值
- 配置项的影响范围
- 调整建议与注意事项
- 环境差异说明

### 2.8 验收标准

- 功能性验收标准（可测试的条件）
- 非功能性验收标准（性能、安全、可用性）
- 验收方式（单元测试/集成测试/手动走查）
- 验收责任方

**缺失任一章节的设计文档不予审批通过。**

## 3. 测试规范

### 3.1 按故事类型的证据表

不同类型的用户故事要求不同的测试证据：

| 故事类型 | 测试方式 | 证据要求 | 强制级别 |
| --- | --- | --- | --- |
| **逻辑**（业务规则、算法、状态转换） | 单元测试 | 覆盖正常/异常/边界路径的单元测试报告 | BLOCKING（必须通过才能合并） |
| **集成**（跨模块、跨服务交互） | 集成测试 | 集成测试报告，覆盖接口契约 | BLOCKING（必须通过才能合并） |
| **UI**（界面展示、交互、视觉） | 手动走查 | 走查清单与截图记录 | ADVISORY（建议但不阻塞） |
| **配置**（环境配置、参数调整） | 冒烟测试 | 冒烟测试通过确认 | ADVISORY（建议但不阻塞） |

### 3.2 自动化测试规则

#### 命名规范

- 测试文件：`[被测模块名].test.[ext]` 或 `[被测模块名].spec.[ext]`
- 测试用例：`should [期望行为] when [前置条件]`
- 示例：`should throw InvalidUserIdError when userId is empty`

#### 确定性

- 测试结果必须稳定可重复
- 禁止依赖当前时间、随机数、网络状态等不确定因素
- 时间相关测试使用 mock 时钟
- 随机数测试使用固定种子

#### 隔离性

- 每个测试独立运行，不依赖其他测试的执行顺序
- 测试之间不共享可变状态
- 每个测试前重置环境（setUp/tearDown）
- 数据库测试使用事务回滚或独立数据库实例

#### 不硬编码

- 禁止硬编码环境相关数据（端口、路径、密钥）
- 测试数据通过工厂模式或 fixture 管理
- 测试配置与生产配置分离

#### 独立性

- 测试不依赖外部服务运行状态
- 外部依赖通过 mock/stub 替代
- 集成测试使用容器化环境保证可重现

## 4. CI/CD 规则

### 4.1 持续集成

- 每次提交自动触发 CI 流水线
- CI 必须包含以下阶段：
  1. 代码静态检查（Lint）
  2. 单元测试
  3. 集成测试
  4. 构建验证
  5. 安全扫描
- CI 失败禁止合并到主干
- CI 配置文件纳入版本控制

### 4.2 持续交付/部署

- 主干合并自动触发部署到预发布环境
- 生产部署需通过发布门禁（`/gate-check`）
- 部署策略：蓝绿部署或滚动部署（由 `devops-lead` 决定）
- 支持快速回滚机制

### 4.3 质量门禁

合并到主干前必须通过以下门禁：

| 门禁 | 检查项 | 负责人 |
| --- | --- | --- |
| 代码审查 | 至少一名审查者 approval | `code-reviewer` |
| 单元测试 | 覆盖率达标，全部通过 | `qa-lead` |
| 集成测试 | 关键路径全部通过 | `qa-lead` |
| 安全扫描 | 无严重漏洞 | `security-lead` |
| 构建验证 | 构建成功，产物完整 | `devops-lead` |

### 4.4 版本管理

- 语义化版本号：`MAJOR.MINOR.PATCH`
- 发布标签格式：`vX.Y.Z`
- 变更日志由 `/changelog` 自动生成
- 补丁说明由 `/patch-notes` 生成
