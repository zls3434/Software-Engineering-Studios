# 上下文管理

本文件定义 Software Engineering Studios 中上下文管理策略，确保 Agent 在有限上下文窗口内高效工作，并支持会话崩溃后的快速恢复。

## 1. 文件即记忆策略

Agent 不依赖对话历史作为主要记忆载体，而是将关键信息持久化到文件系统中。

**核心原则：**
- 任何需要在会话间保留的信息必须写入文件
- 文件路径作为上下文引用，而非完整内容
- Agent 通过读取文件恢复上下文，而非依赖对话回溯
- 长期决策记录在架构文档和 ADR 中
- 短期任务状态记录在会话状态文件中

**优势：**
- 不受上下文窗口大小限制
- 支持多 Agent 共享同一上下文
- 会话崩溃后可快速恢复
- 便于审计和追溯

## 2. 会话状态文件 active.md

每个活跃会话维护一个状态文件 `production/session-state/active.md`，作为当前工作的核心上下文。

### 2.1 文件结构

```markdown
# 活跃会话状态

## STATUS
- 当前阶段：[阶段名称]
- 当前任务：[任务描述]
- 负责 Agent：[Agent 名称]
- 最后更新：[时间戳]

## 当前任务
- 任务 ID：[ID]
- 描述：[详细描述]
- 进度：[百分比或步骤]
- 阻塞项：[BLOCKED 原因，无则填"无"]

## 文件列表
- 本次会话创建/修改的文件清单

## 架构决策
- 本次会话作出的架构决策摘要

## Sprint 状态
- 当前 Sprint：[编号]
- 进行中故事：[列表]
- 已完成故事：[列表]

## Agent 调用
- 本次会话调用的子 Agent 记录

## 测试结果
- 最近的测试执行结果摘要

## 阻塞项
- 当前所有阻塞项及处理状态
```

### 2.2 更新规则

- 每完成一个子任务后更新 `active.md`
- 每次 Agent 调用前后更新调用记录
- 遇到 BLOCKED 时立即记录
- 会话结束前归档至 `production/session-logs/`

## 3. 状态行 STATUS 块

`active.md` 顶部的 STATUS 块是快速定位当前工作状态的紧凑区域。

**用途：**
- 任何 Agent 读取 `active.md` 时首先查看 STATUS 块
- 包含最小必要信息以判断当前进度
- 支持快速决定从何处恢复

**规则：**
- STATUS 块必须始终位于文件顶部
- 保持在 10 行以内
- 字段值简洁，详细信息放在下方对应章节
- 每次更新 `active.md` 时同步刷新 STATUS 块

## 4. 增量文件写入

对于长文档或复杂产出物，采用增量写入策略，避免单次生成导致上下文溢出。

### 4.1 写入流程

```
1. 创建骨架 → 2. 逐章填充 → 3. 写入文件 → 4. 更新状态
```

1. **创建骨架**：先定义文档大纲（章节标题和占位符）
2. **逐章填充**：每次专注一个章节，生成完整内容
3. **写入文件**：将完成的章节写入文件
4. **更新状态**：在 `active.md` 中记录已完成章节

### 4.2 示例

```
创建 product-concept.md：
  骨架：概述 → 目标用户 → 核心功能 → 竞品分析 → 商业模式
  第1步：写入"概述"章节 → 更新 active.md（已完成 1/5）
  第2步：写入"目标用户"章节 → 更新 active.md（已完成 2/5）
  ...
```

## 5. 主动压缩

当上下文使用达到阈值时，主动进行压缩以释放空间。

### 5.1 压缩阈值

- **60-70% 压缩**：当上下文使用达到 60% 时开始准备压缩，达到 70% 时立即执行
- 通过 `pre-compact.sh` Hook 在压缩前保存进度
- 通过 `post-compact.sh` Hook 在压缩后恢复提醒

### 5.2 聚焦压缩

压缩时保留关键信息，丢弃冗余内容：

**保留：**
- `active.md` 引用
- 文件列表与路径
- 架构决策记录
- Sprint 状态
- Agent 调用历史
- 测试结果摘要
- 阻塞项清单
- 当前任务描述

**丢弃：**
- 已完成任务的详细对话过程
- 已写入文件的完整内容（通过路径引用即可）
- 非关键的中间探索过程
- 冗余的确认与回复

## 6. 按任务类型的上下文预算

根据任务复杂度分配上下文预算，避免单一任务消耗过多资源：

| 任务类型 | 上下文预算 | 典型任务 | 策略 |
| --- | --- | --- | --- |
| **轻量** | ~3k tokens | 状态查询、简单格式化、文件检查 | 直接执行，无需子 Agent |
| **中等** | ~8k tokens | 单模块开发、文档撰写、单文件审查 | 主 Agent 直接处理或派发单个子 Agent |
| **重量** | ~15k tokens | 跨模块设计、全量审查、架构决策 | 委派子 Agent，主 Agent 仅协调 |

**预算超限处理：**
- 轻量任务超限 → 重新评估是否应升级为中等任务
- 中等任务超限 → 拆分为多个轻量子任务
- 重量任务超限 → 委派更多子 Agent 分摊上下文

## 7. 子 Agent 委派

通过委派子 Agent 实现上下文分摊和并行执行。

### 7.1 委派原则

- 子 Agent 在独立上下文中运行
- 主 Agent 通过任务描述和文件引用传递上下文
- 子 Agent 返回精炼结果而非完整日志
- 主 Agent 负责整合多个子 Agent 的结果

### 7.2 委派时机

- 任务上下文需求超过主 Agent 预算
- 任务可独立完成，无需主 Agent 介入
- 多个同类任务可并行执行
- 需要深度分析且避免主 Agent 上下文污染

### 7.3 委派协议

```
主 Agent → Task 工具 → 派发子 Agent
  任务描述：[清晰的目标]
  文件引用：[需要读取的文件路径]
  预期产出：[产出物类型和路径]
  模型层级：[Haiku/Sonnet/Opus]
子 Agent → 执行 → 返回结果
主 Agent → 收集 → 整合 → 继续
```

## 8. 压缩指令

当执行上下文压缩时，必须遵循以下保留清单：

### 必须保留

1. **`active.md` 引用** — 文件路径和 STATUS 块内容
2. **文件列表** — 本会话创建/修改的所有文件路径
3. **架构决策** — 本次会话作出的关键架构决策摘要
4. **Sprint 状态** — 当前 Sprint 编号和进行中故事
5. **Agent 调用** — 已调用的子 Agent 名称和任务
6. **测试结果** — 最近的测试通过/失败状态
7. **阻塞项** — 所有未解决的阻塞项
8. **当前任务** — 正在执行的任务描述和进度

### 可以丢弃

1. 已写入文件的完整内容（通过路径引用）
2. 已完成任务的中间探索对话
3. 非关键的确认回复
4. 文件的完整内容引用（保留路径即可）

## 9. 会话崩溃恢复

当会话意外中断或崩溃时，按以下步骤恢复：

### 9.1 恢复流程

```
读取 active.md → 读取部分完成文件 → 从断点继续
```

**步骤 1：读取 active.md**
- 查看 STATUS 块了解当前阶段和任务
- 查看"当前任务"了解具体进度
- 查看"阻塞项"了解未解决问题
- 查看"文件列表"了解已处理文件

**步骤 2：读取部分完成文件**
- 检查 `active.md` 中记录的文件是否完整
- 对于增量写入的文件，检查已完成章节
- 验证文件内容与状态记录一致

**步骤 3：从断点继续**
- 根据进度记录确定下一步操作
- 若文件部分完成，继续未完成章节
- 若任务被阻塞，优先处理阻塞项
- 更新 `active.md` 记录恢复时间

### 9.2 恢复检查清单

- [ ] `active.md` 存在且可读
- [ ] STATUS 块反映最新状态
- [ ] 部分完成文件已验证
- [ ] 阻塞项已识别
- [ ] 下一步操作已明确
- [ ] `active.md` 已更新恢复记录

### 9.3 无 active.md 的处理

若 `active.md` 不存在或已归档：
1. 运行 `/project-stage-detect` 检测当前项目阶段
2. 检查 `production/session-logs/` 中的历史会话记录
3. 根据项目文件状态推断进度
4. 创建新的 `active.md`
