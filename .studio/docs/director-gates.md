# 总监门禁定义

本文件定义 Software Engineering Studios 中总监层的 5 个门禁（Gate）。门禁是总监层 Agent 对关键阶段产出物的质量把关机制，必须在进入下一阶段前通过对应门禁。

## 1. 门禁概览

| 门禁 ID | 负责总监 | 用途 | 判定标准 |
| --- | --- | --- | --- |
| PD-PILLARS | 产品总监 | 验证产品支柱 | APPROVE / CONCERNS / REJECT |
| PD-CONCEPT-FIT | 产品总监 | 产品概念对齐 | APPROVE / CONCERNS / REJECT |
| CA-ARCHITECTURE | 首席架构师 | 验证架构 | APPROVE / CONCERNS / REJECT |
| CA-FEASIBILITY | 首席架构师 | 技术可行性评估 | APPROVE / CONCERNS / REJECT |
| PM-SCOPE | 项目经理 | 范围评估 | APPROVE / CONCERNS / REJECT |

**门禁判定含义：**

- **APPROVE**：通过门禁，可进入下一阶段
- **CONCERNS**：有保留意见，可在修复后继续，但需记录关注项
- **REJECT**：不通过，必须修复后重新提交门禁审查

## 2. PD-PILLARS（产品支柱验证）

**负责总监：** `product-director`（产品总监）

### 触发条件

- 产品概念文档生成后
- 产品支柱定义完成时
- 产品方向发生重大调整时

### 传入数据

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| 支柱集 | 列表 | 产品支柱定义列表，每个支柱含名称、描述、成功指标 |
| 设计测试 | 文档 | 针对支柱设计的验证测试方案 |

### 判定标准

| 判定 | 条件 |
| --- | --- |
| **APPROVE** | 支柱定义清晰、可度量、与产品愿景一致，设计测试可验证支柱达成 |
| **CONCERNS** | 支柱定义基本清晰但部分指标不够可度量，或设计测试覆盖不全 |
| **REJECT** | 支柱定义模糊、不可度量、与产品愿景不一致，或缺失设计测试 |

### 输出格式

```markdown
## 门禁审查报告：PD-PILLARS

- 审查时间：[时间戳]
- 审查者：product-director
- 判定结果：[APPROVE / CONCERNS / REJECT]

### 支柱逐项评估

| 支柱名称 | 评估 | 说明 |
| --- | --- | --- |
| [支柱1] | [通过/关注/否决] | [详细说明] |
| [支柱2] | [通过/关注/否决] | [详细说明] |

### 设计测试评估

- 覆盖性：[评估]
- 可度量性：[评估]
- 可执行性：[评估]

### 关注项 / 修复项

1. [关注项描述及建议]
2. [关注项描述及建议]

### 结论

[判定理由与后续行动建议]
```

## 7. 门禁执行流程

1. **触发门禁**：对应阶段产出物就绪后，由负责 Agent 调用门禁检查
2. **传入数据**：收集所需传入数据并提交至总监 Agent
3. **总监审查**：总监 Agent 对照判定标准进行审查
4. **输出报告**：生成门禁审查报告并保存至 `production/` 对应目录
5. **判定处理**：
   - APPROVE → 继续下一阶段
   - CONCERNS → 记录关注项，修复后继续
   - REJECT → 修复后重新提交门禁审查
6. **更新状态**：在 `active.md` 中记录门禁结果

## 8. 门禁与阶段映射

| 阶段 | 对应门禁 |
| --- | --- |
| 需求阶段 | PD-PILLARS、PD-CONCEPT-FIT |
| 架构设计阶段 | CA-ARCHITECTURE、CA-FEASIBILITY |
| 预开发阶段 | PM-SCOPE |
| 发布阶段 | gate-check（由 `/gate-check` 命令触发，综合所有门禁结果） |
