# 审查工作流

本文件定义 Software Engineering Studios 中的审查工作流，包括审查模式、各阶段关键审查点和标准审查流程。

## 1. 审查模式

审查模式通过 `production/review-mode.txt` 文件配置，控制审查的严格程度和覆盖范围。

### 1.1 三种审查模式

| 模式 | 说明 | 适用场景 | 审查严格度 |
| --- | --- | --- | --- |
| **full**（完整审查） | 对所有产出物进行完整审查，涵盖所有审查点 | 新项目、关键里程碑、高风险变更 | 最高 |
| **lean**（精简审查） | 聚焦关键审查点，跳过可选审查 | 迭代开发、低风险变更、时间紧迫 | 中等 |
| **solo**（单人审查） | 由单一 Agent 完成所有审查，减少协调开销 | 个人项目、原型开发、快速验证 | 最低 |

### 1.2 模式配置

在 `production/review-mode.txt` 中写入模式名称：

```
full
```

或

```
lean
```

或

```
solo
```

**配置规则：**
- 文件不存在时默认使用 `full` 模式
- 模式变更需通知所有相关 Agent
- `solo` 模式下跨领域审查由主 Agent 自行完成
- `lean` 模式跳过所有标记为"可选"的审查点

### 1.3 模式对比

| 维度 | full | lean | solo |
| --- | --- | --- | --- |
| 审查 Agent 数量 | 多 Agent 协同 | 关键 Agent 参与 | 单一 Agent |
| 审查覆盖 | 所有审查点 | 必需审查点 | 主 Agent 判断 |
| 审查深度 | 完整对照标准 | 聚焦关键标准 | 简化检查 |
| 审查耗时 | 最长 | 中等 | 最短 |
| 适用风险等级 | 高 | 中 | 低 |

## 2. 每阶段关键审查点

### 2.1 需求阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `design-review` | `/design-review` | full/lean 必需，solo 可选 | 需求文档完整性、一致性、可测试性 |

**审查重点：**
- 需求规格是否包含 8 个必需章节
- 用户故事是否清晰且可验收
- 需求之间是否存在矛盾
- 需求是否可追溯到产品概念
- 边界情况是否覆盖

### 2.2 架构设计阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `architecture-review` | `/architecture-review` | full/lean/solo 均必需 | 架构合理性、ADR 充分性、技术选型 |

**审查重点：**
- 架构文档是否完整
- ADR 是否至少 3 条且论证充分
- 技术选型是否与需求匹配
- 模块划分是否清晰
- 是否考虑可扩展性和可维护性
- 是否存在架构反模式

### 2.3 技术设计阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `ux-review` | `/ux-review` | full/lean 必需，solo 可选 | UX 设计一致性、可用性、无障碍性 |

**审查重点：**
- UX 设计是否符合设计系统
- 交互流程是否流畅
- 无障碍需求是否满足
- 响应式设计是否覆盖
- UX 设计数量是否满足最低要求（至少 3 个）

### 2.4 预开发阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `gate-check` | `/gate-check` | full/lean 必需，solo 可选 | 门禁检查、Sprint 规划合理性 |

**审查重点：**
- 史诗和故事是否完整
- Sprint 规划是否合理
- 工作量估算是否准确
- PM-SCOPE 门禁是否通过
- 团队资源是否匹配

### 2.5 开发阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `code-review` | `/code-review` | full/lean 必需，solo 可选 | 代码质量、规范遵循、测试覆盖 |

**审查重点：**
- 代码是否符合编码规范
- 公共 API 是否有文档注释
- 是否硬编码配置
- 依赖注入是否优于单例
- 提交信息是否符合 Conventional Commits
- 测试证据是否满足故事类型要求
- 是否存在禁止模式

### 2.6 测试优化阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `test-evidence-review` | `/test-evidence-review` | full 必需，lean/solo 可选 | 测试证据完整性、覆盖充分性 |

**审查重点：**
- 测试证据是否可追溯至需求
- 逻辑故事是否有单元测试（BLOCKING）
- 集成故事是否有集成测试（BLOCKING）
- UI 故事是否有手动走查记录（ADVISORY）
- 配置故事是否有冒烟测试（ADVISORY）
- 测试是否满足确定性、隔离性、独立性要求
- 冒烟测试是否通过

### 2.7 发布阶段

| 审查点 | 命令 | 模式要求 | 审查内容 |
| --- | --- | --- | --- |
| `launch-checklist` | `/launch-checklist` | full/lean/solo 均必需 | 上线准备、回滚方案、监控配置 |

**审查重点：**
- 发布检查清单是否全部完成
- 上线步骤是否清晰
- 回滚方案是否可行
- 监控和告警是否配置
- 变更日志是否生成
- 安全审计是否完成（full 模式）
- 性能是否达标（full 模式）

## 3. 标准审查流程

所有审查遵循统一的四步流程：

### 步骤 1：读取文档

- 读取待审查的产出物文件
- 读取对应审查标准（本文件或相关规范文件）
- 读取相关上下文（需求、架构决策等）

### 步骤 2：对照标准

- 逐项对照审查标准检查产出物
- 标记每项检查结果：通过 / 关注 / 不通过
- 记录不符合项的具体位置和原因

### 步骤 3：生成报告

- 按统一格式生成审查报告
- 报告包含：审查对象、审查标准、逐项结果、总体判定、修复建议
- 报告保存至对应目录

### 步骤 4：建议修改

- 列出所有需修复项及优先级
- 建议具体修改方案
- 标注 BLOCKING（阻塞）和 ADVISORY（建议）项
- 将修复任务分配至对应 Agent

### 审查报告模板

```markdown
## 审查报告：[审查点名称]

- 审查时间：[时间戳]
- 审查者：[Agent 名称]
- 审查模式：[full/lean/solo]
- 审查对象：[文件路径列表]

### 审查标准

[本次审查依据的标准列表]

### 逐项检查结果

| 检查项 | 结果 | 说明 |
| --- | --- | --- |
| [检查项1] | [通过/关注/不通过] | [说明] |
| [检查项2] | [通过/关注/不通过] | [说明] |

### 总体判定

- 判定：[APPROVE / CONCERNS / REJECT]
- BLOCKING 项数量：[数量]
- ADVISORY 项数量：[数量]

### 修复建议

#### BLOCKING（必须修复）

1. [修复项及建议]
2. [修复项及建议]

#### ADVISORY（建议修复）

1. [修复项及建议]
2. [修复项及建议]

### 后续行动

- [ ] 修复 BLOCKING 项
- [ ] 重新提交审查
- [ ] 记录 ADVISORY 项至技术债务清单
```

## 4. 审查与门禁的关系

- **门禁**（详见 `director-gates.md`）由总监层 Agent 执行，是阶段间的硬性把关
- **审查**由部门负责人或专家 Agent 执行，是阶段内的质量检查
- 门禁判定可参考审查报告
- 审查不通过可能导致门禁判定为 CONCERNS 或 REJECT

### 4.1 资产审查

资产审查是针对动态扩展机制新增资产（Agent/Skill/Rule）的专项审查：

- **触发命令**：`/asset-review [资产类型] [资产名称]`
- **执行者**：chief-architect（对应 CA-ASSET 门禁）
- **审查对象**：通过 `/create-agent`、`/create-skill` 或 `/create-rule` 创建的资产草案
- **审查标准**：依据资产类型执行对应检查清单（详见 `docs/extension-mechanism.md` 第 6 节）
- **判定标准**：APPROVE / CONCERNS / REJECT（与门禁判定一致）
- **报告归档**：`.studio/registry/reviews/`
- **完整规范**：详见 `docs/extension-mechanism.md`

资产审查与常规审查的区别：资产审查聚焦于新增资产本身的合规性和必要性，常规审查聚焦于阶段产出物的质量。
