# 代码风格指南

本文件从 `docs/coding-standards.md` 和 `docs/technical-preferences.md` 浓缩，是平台无关的代码风格规范源。

## 1. 编码原则

### 1.1 公共 API 文档注释

所有公共 API 必须包含完整的文档注释：

- 每个公共函数/方法/类必须包含用途说明
- 参数必须标注类型、含义和约束
- 返回值必须标注类型和含义
- 异常/错误必须列出可能的类型和触发条件
- 示例代码应附在复杂 API 的注释中

### 1.2 配置数据驱动不硬编码

所有环境相关配置、业务参数、阈值等必须数据驱动：

- 配置文件存放于 `src/config/` 或 `config/` 目录
- 环境变量通过 `.env` 文件管理，禁止提交到版本控制
- 业务可配置参数应支持运行时调整
- 不同环境（开发/测试/预发布/生产）使用独立配置

### 1.3 依赖注入优于单例

优先使用依赖注入（DI）模式：

- 通过构造函数注入依赖
- 使用 IoC 容器管理依赖生命周期
- 单例仅在无状态工具类、基础设施层或经 `chief-architect` 审批时允许

### 1.4 验证驱动开发

每个功能在实现前必须明确验证方式：

- 每个用户故事必须包含验收标准
- 实现前先定义测试用例
- 测试证据必须可追溯至需求

## 2. 命名约定

### 2.1 文件命名

| 类型 | 约定 | 示例 |
|------|------|------|
| 前端组件 | PascalCase | `UserProfile.tsx` |
| 前端样式 | kebab-case | `user-profile.css` |
| 后端模块（Python） | snake_case | `user_service.py` |
| 后端模块（Java/Go） | PascalCase/camelCase | `UserService.java` |
| 测试文件 | `[被测名].test.[ext]` 或 `[被测名].spec.[ext]` | `auth.test.ts` |
| 文档文件 | kebab-case | `quick-start.md` |
| ADR | `ADR-NNN-标题.md` | `ADR-001-选择-PostgreSQL.md` |

### 2.2 代码命名

| 类型 | 约定 | 适用语言 |
|------|------|----------|
| 变量与函数 | camelCase | JavaScript/TypeScript/Java/Go |
| 变量与函数 | snake_case | Python |
| 类与接口 | PascalCase | 所有语言 |
| 常量 | UPPER_SNAKE_CASE | 所有语言 |
| 枚举 | PascalCase 类型名，UPPER_SNAKE_CASE 值 | 所有语言 |

### 2.3 Git 分支命名

采用基于主干的开发模式：

| 分支类型 | 格式 | 说明 |
|----------|------|------|
| 主干 | `main` | 生产分支 |
| 功能分支 | `feature/[简短描述]` | 新功能开发 |
| 修复分支 | `fix/[简短描述]` | 缺陷修复 |
| 热修复分支 | `hotfix/[简短描述]` | 生产紧急修复 |
| 发布分支 | `release/vX.Y.Z` | 发布准备 |

## 3. 禁止模式

以下模式在所有代码中严格禁止。

### 3.1 通用禁止

- **禁止硬编码**环境相关配置（数据库连接、API 密钥、端口等）
- **禁止 `any` 类型**（TypeScript）或跳过类型检查
- **禁止 `console.log`** 或等效调试输出在生产代码中使用
- **禁止空 catch 块**（忽略错误）
- **禁止使用已弃用的 API 或依赖**

### 3.2 前端禁止

- 禁止直接操作 DOM（框架提供抽象层时）
- 禁止在组件中直接调用全局状态（应通过依赖注入或 props）
- 禁止内联样式（应使用样式表或 CSS-in-JS 方案）

### 3.3 后端禁止

- 禁止在业务逻辑中直接拼接 SQL（应使用参数化查询或 ORM）
- 禁止在请求处理中执行阻塞 I/O（Node.js 场景）
- 禁止跨请求共享可变状态（除非通过显式共享存储）

### 3.4 数据库禁止

- 禁止在生产环境直接执行破坏性操作（DROP/DELETE/TRUNCATE）
- 禁止创建无索引的外键关联
- 禁止存储明文敏感数据（密码、令牌等）

## 4. 设计文档规范

每个系统/模块的设计文档必须包含以下 8 个必需章节：

1. **概述** — 系统名称、定位、核心职责、架构位置
2. **用户故事** — 以用户视角描述使用场景
3. **详细规则** — 业务规则、状态转换、权限控制、数据校验
4. **数据模型** — 实体定义、字段类型、索引策略、迁移方案
5. **边界情况** — 极端输入、并发竞态、失败回滚、性能边界
6. **依赖关系** — 上游/下游依赖、外部服务、版本兼容性
7. **可配置参数** — 配置项、默认值、影响范围、环境差异
8. **验收标准** — 功能性/非功能性标准、验收方式、责任方

> 缺失任一章节的设计文档不予审批通过。

## 5. Conventional Commits

所有 Git 提交信息必须遵循 Conventional Commits 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(auth): 添加 OAuth2 登录支持` |
| `fix` | 缺陷修复 | `fix(api): 修复分页参数 off-by-one 错误` |
| `docs` | 文档变更 | `docs(api): 更新用户接口文档` |
| `refactor` | 重构（不改变外部行为） | `refactor(auth): 提取令牌验证逻辑` |
| `test` | 测试相关 | `test(user): 添加用户注册单元测试` |
| `chore` | 构建/工具/杂项 | `chore(deps): 升级依赖至最新版本` |

**规则：**

- `subject` 使用简体中文，不超过 50 字
- `body` 说明变更原因和背景
- `footer` 标注 BREAKING CHANGE 或关联 issue
- 提交前由 `validate-commit.sh` Hook 自动校验
