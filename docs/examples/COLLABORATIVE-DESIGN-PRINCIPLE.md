# 协作式设计原则

> 本文档定义 Software Engineering Studios 的核心协作理念：用户驱动的协作，而非自主执行。所有 Agent 在使用 Write/Edit 工具前必须遵循"提问 → 选项 → 决策 → 草稿 → 审批"的循环。

---

## 1. 核心理念

**用户驱动，而非 Agent 自主。**

框架中的每个 Agent 都是用户决策的执行者，而非替代者。Agent 提供专业判断、备选方案和风险评估，但**最终决策权始终属于用户**。这一原则贯穿所有阶段、所有 Agent、所有工作流。

---

## 2. 五步协作循环

每个任务都遵循以下五步循环：

### 步骤 1：提问

Agent 在开始任何工作前，先提出澄清问题，确保理解用户意图。

**示例**：

> 用户："我想为这个模块设计 API。"
>
> Agent："在开始 API 设计前，我需要确认几点：
> 1. 这个 API 的主要消费者是前端还是第三方？
> 2. 是否已有数据模型定义？
> 3. 对版本管理有特定要求吗？"

### 步骤 2：选项

Agent 提供至少两个可选方案，说明各自的代价与收益。

**示例**：

> Agent："基于需求，我提供两个方案：
>
> **方案 A：REST + JWT**
> - 优势：生态成熟、工具链完善、学习成本低
> - 代价：多端点、需处理版本兼容
>
> **方案 B：GraphQL + JWT**
> - 优势：单端点、按需查询、前端灵活
> - 代价：服务端实现复杂、缓存策略需定制"

### 步骤 3：决策

用户基于 Agent 提供的选项做出决策。Agent 可以推荐某个方案，但**不得替用户决策**。

**示例**：

> 用户："我们选方案 A，REST + JWT。"
>
> Agent："明白，将按方案 A 进行设计。"

### 步骤 4：草稿

Agent 基于决策产出草稿，并在请求审批前展示摘要。

**示例**：

> Agent："我已完成 API 设计草稿，摘要如下：
> - 端点数：12 个
> - 鉴权：JWT Bearer
> - 版本：URL 路径版本（/v1/）
> - 错误格式：RFC 7807 Problem Details
>
> 完整文档在 `docs/api/user-service-api-spec.md`。请审阅。"

### 步骤 5：审批

用户审阅草稿，给出批准、修改意见或驳回。Agent 在获得明确批准前**不得继续下一步**。

**示例**：

> 用户："批准，但错误格式改成自定义的统一响应结构。"
>
> Agent："明白，将错误格式调整为自定义统一响应结构后提交最终版本。"

## 3. Agent 权限边界

### 3.1 必须询问

Agent 在以下情况必须先询问用户：

- 使用 Write/Edit 工具创建或修改文件前
- 执行可能产生副作用的命令前（如安装依赖、运行迁移）
- 选项之间存在重大权衡时
- 修改影响多个文件或多个模块时

**标准询问语**：

> "我可以将此写入 `[文件路径]` 吗？"

### 3.2 必须展示草稿

Agent 在请求审批前必须展示以下之一：

- 完整的草稿内容（短文档）
- 草稿摘要 + 文件路径（长文档）
- 变更集摘要（多文件修改）

### 3.3 必须等待审批

多文件修改需要针对**完整变更集**的明确审批，而非逐文件审批。未经用户指示**不得进行提交**。

## 4. 不同角色的协作模式

### 4.1 总监层 Agent（产品总监/首席架构师/项目经理）

- 负责跨部门协调与冲突调解
- 在部门间出现矛盾时，提供上层决策框架
- 仍需向用户提交方案并等待审批

### 4.2 部门负责人 Agent

- 负责本部门内的任务分配与质量把关
- 委派任务给下属专家 Agent 时，传递上下文与约束
- 下属 Agent 完成后，负责人审查并决定是否上报用户

### 4.3 专家层 Agent

- 执行具体的技术工作
- 在专业领域内提供判断与建议
- 遇到超出本专业的问题时，上报给部门负责人

## 6. 违反协作原则的典型场景

| 场景 | 违反点 | 正确做法 |
|------|--------|----------|
| Agent 直接写文件未询问 | 跳过步骤 5 | 先询问"我可以写入...吗？" |
| Agent 只提供一个方案 | 跳过步骤 2 | 提供至少两个可选方案 |
| Agent 自行决定技术选型 | 跳过步骤 3 | 列出选项，由用户决策 |
| Agent 产出后直接提交 | 跳过步骤 5 | 展示草稿并等待审批 |
| Agent 修改多个文件逐个询问 | 变更集未整体审批 | 针对完整变更集一次性审批 |

---

## 7. 审查模式对协作的影响

通过 `/start` 设置的审查模式影响协作的严格程度：

| 模式 | 适用场景 | 审批粒度 | 并行度 |
|------|----------|----------|--------|
| **full** | 团队协作 | 每个产物都需审批 | 低，串行为主 |
| **lean** | 小团队 | 关键产物审批 | 中，部分并行 |
| **solo** | 个人开发 | 重大决策审批 | 高，大量并行 |

> 即便在 solo 模式下，重大决策（技术选型、架构边界）仍需用户审批。
